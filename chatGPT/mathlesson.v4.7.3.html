    Authoring Guide

    MathLesson v3.9.12 ŌĆö didactic manual for planning, editing, exporting, and testing single-file mathematical lessons.

    1. What MathLesson Is

    MathLesson is a single-file mathematical lesson workbench. A lesson is rendered from a structured JavaScript object plus HTML section content. It supports prose, LaTeX, callouts, interactive plots, tables, diagrams, geometry boards, video slots, SVG, quizzes, notes, export/import, and local persistence.

    Mental model

    A MathLesson lesson is a finite ordered tuple

    \[ L=(T,A,S,Q,R) \]

    where \(T\) is the title, \(A\) is audio metadata, \(S\) is the ordered section list, \(Q\) is the quiz list, and \(R\) is any populated legacy registry such as plots or tables.

    2. Expected Prerequisites

    SkillNeeded forMinimum level

    3. The Canonical Lesson JSON Shape

    Exports use schema version mathlesson.lesson.v1. The exported object is intentionally explicit: section HTML is resolved, legacy empty registries are omitted, and discovered blocks are listed in blockInventory.

    {
  "schemaVersion": "mathlesson.lesson.v1",
  "exportedFrom": "mathlesson.v3.7.8",
  "title": "Example Lesson",
  "audio": { "src": "", "label": "Narration" },
  "sections": [
    {
      "id": "sec-1",
      "label": "┬¦1 Introduction",
      "content": "<h1>Introduction</h1>..."
    }
  ],
  "quiz": [
    {
      "type": "mc",
      "stem": "Which expression equals $1$?",
      "options": ["$e^{0}$", "$0$", "$-1$", "$\pi$"],
      "answer": 0,
      "feedback": "$e^0=1$."
    }
  ],
  "blockInventory": []
}

    4. Section Structure

    A section needs an id, a sidebar label, and HTML content. Built-in lessons may use contentId internally, but exported lessons resolve this to plain content.

    {
  "id": "normal-form",
  "label": "┬¦2 Normal Form",
  "content": "<h1>Normal Form</h1><p>Text with $a_x$.</p>"
}
    Rule

    Author sections as ordinary HTML. Use the Insert toolbar for common block templates, then use the Code Blocks tab for focused editing of Plotly, Mermaid, JSXGraph, Arquero, Manim, and SVG blocks.

    Calculator Panel

    The Calculator panel combines Nerdamer for symbolic algebra and math.js for numerical computation. It can evaluate, simplify, expand, factor, differentiate, integrate, solve, substitute values, and convert output to LaTeX.

    Calculator results can be inserted into the currently selected Lesson Editor section as inline math, display math, a worked step, or a callout.

    Statistics & Probability Panel

    The Statistics panel combines simple-statistics and jStat where available. It supports descriptive statistics, distribution PDF/CDF/quantile calculations, seeded simulations, simple linear regression, and basic life-table/survival blocks.

    Reusable statistics blocks use <div class="stat-block" data-stat-type="..." data-stat-config="..."></div> and are detected by blockInventory.

    5. Math Syntax

    SyntaxUse
    $...$Inline math
    $$...$$Display math
    \(...\)Inline math
    \[...\]Display math

    Literal dollar amounts should be escaped as \$24{,}000 when they appear in content that KaTeX scans.

    8. Plotly / Explore Widgets

    Explore widgets are preferred over legacy plots registries for new authoring.

    <div class="explore-widget" data-explore="2d"
  data-title="Sine family"
  data-params='[
    {"id":"A","label":"Amplitude","min":0.1,"max":3,"step":0.1,"value":1},
    {"id":"f","label":"Frequency","min":0.5,"max":5,"step":0.5,"value":1}
  ]'
  data-fn="(x,p) => p.A * Math.sin(2*Math.PI*p.f*x)"
  data-xrange="[-2,2]"
  data-xlabel="x" data-ylabel="y"
  style="height:260px;">
</div>

    9. Arquero Blocks

    Arquero blocks execute JavaScript that returns HTML, usually by calling renderAqTable(tbl, caption).

    <div class="aq-wrap" data-aq-title="Grouped summary">
  <script type="text/arquero">
const raw = [
  {group:"A", value:10},
  {group:"A", value:14},
  {group:"B", value:20}
];
const tbl = aq.from(raw)
  .groupby("group")
  .rollup({ avg_value: d => aq.op.mean(d.value) });
return renderAqTable(tbl, "Grouped summary");
  </script>
</div>

    10. Mermaid Blocks

    Mermaid blocks are supported. In a few browser/file situations, a Mermaid diagram may fail if the diagram text is too complicated. When that happens, the lesson should either simplify the Mermaid text or use a simple inline SVG drawing as a backup picture. An SVG drawing is ordinary HTML-based drawing code that the browser can display without Mermaid.

    <div class="mermaid-wrap">
  <div class="mermaid-label">Dependency graph</div>
  <pre class="mermaid">
flowchart TD
    A[Input] --> B[Transform]
    B --> C[Output]
  </pre>
</div>
    Mermaid stability rule

    Use simple node IDs and simple labels first. A node ID is the short internal name used by Mermaid, such as A, B, or step1. A label is the text shown to the reader, such as Input or Output. Avoid punctuation-heavy labels, raw LaTeX, unusual arrows, or complex symbols inside Mermaid nodes. If a Mermaid diagram fails, simplify the Mermaid source or replace the diagram with a plain HTML/SVG drawing.

    11. JSXGraph Blocks

    A JSXGraph block declares a board container and either a named initializer or inline code in a text/jsxgraph script.

    <div class="jsx-wrap" data-jsx="line-demo" data-title="Line demo" style="height:300px;">
  <script type="text/jsxgraph">
board.suspendUpdate();
const A = board.create('point', [-2, 1], {name:'A'});
const B = board.create('point', [ 2, 1], {name:'B'});
board.create('line', [A, B], {strokeColor:'#5ecbff', strokeWidth:2});
board.unsuspendUpdate();
  </script>
</div>

    12. Manim Video Blocks

    MathLesson does not render Manim scenes. It displays videos or placeholders produced by Manim CE.

    <div class="manim-slot">

    13. SVG Blocks

    Use SVG for stable diagrams, backup drawings, static mathematical illustrations, and export-friendly visuals. A backup drawing is a simple diagram that still displays if an interactive or generated diagram does not render.

    16. Code Blocks Editor

    The Code Blocks tab scans the current lesson section content for supported code-bearing blocks. Use it for focused editing rather than searching through large HTML strings.

    1. Select a section in the Sections tab.
    2. Switch to Code Blocks.
    3. Filter by block type if needed.
    4. Edit, validate, save, duplicate, or delete the block.

    17. Export / Import

    Use Export Editor JSON for the lesson currently loaded in the editor. Use Export Active Lesson for the lesson currently open in the main viewer, including built-in lessons.

    Export invariants
    • Exports include schemaVersion and exportedFrom.
    • Built-in template sections export as resolved sections[i].content.
    • Empty legacy plots and tables are omitted.
    • blockInventory lists discovered blocks in sections and rich quiz stems.

    18. Legacy Registries vs Inline Blocks

    Older built-in lessons may still contain populated legacy registries:

    plots: {
  "beta-plot": (L, el) => { ... }
}

tables: {
  "mortality-table-wrap": () => { ... }
}

    New authoring should prefer inline blocks in sections[i].content. Legacy registries are preserved for compatibility but are not the future canonical model.

    19. blockInventory

    blockInventory is an export diagnostic and pedagogical index. It tells you what blocks were discovered and where they live.

  "questionIndex": 1,
  "storage": "quiz[1].stem",
  "source": "quiz-inline-explore-widget"
}

    20. Reveal Export Planner

    The Reveal Export Planner should be used before exporting a lesson as a Reveal.js deck. It evaluates each section as authored, estimates whether it will fit on a slide, and proposes a concrete slide plan.

    Reference ŌåÆ Reveal Export Planner ŌåÆ Scan Active Lesson

    The planner reports:

    • Risk: whether a section is likely to overflow in Reveal.
    • Proposed slides: how the section will be split into Reveal slides.
    • Recommended manual breaks: exact places where the author should insert <!-- slide -->.

    Use explicit authored slide breaks whenever the planner identifies a section as high risk:

    <!-- slide -->

    or:

    <hr class="slide-break">

    Put large visuals, interactive widgets, long derivations, and major theorem/example transitions on their own Reveal slides. The exporter may auto-split oversized sections, but deliberate authored breaks produce more predictable decks.

    21. Print/PDF Planner

    The Print/PDF Planner is separate from the web lesson view because browser layout and printed/PDF layout use different page geometries. Use it before saving to PDF.

    Reference ŌåÆ Print/PDF Planner ŌåÆ Scan Active Lesson

    Recommended print controls:

    • print-full-width: widen a graph, board, table, SVG, or diagram for print.
    • print-graph-page: force a graph or visual block onto its own printed page.
    • print-compact: reduce density for a specific block.
    • print-landscape: flag a section that should be printed using landscape geometry.
    • avoid-break: prevent a small block from splitting.
    • allow-break: allow a long table or long prose block to split.

    The default geometry is PDF portrait. Use landscape only for wide graphs or geometry boards. The print planner also adds page margins plus top, bottom, left, and right section padding so content does not run to the edge of the paper.

    22. Recommended Authoring Workflow

    1. Draft the lesson outline first: title, sections, quiz objectives.
    2. Add plain section prose and LaTeX.
    3. Add callouts for definitions, theorems, examples, and exercises.
    4. Add one interactive block at a time.
    5. Use Preview after each structural change.
    6. Use Simple Edit for prose, math, and ordinary callouts.
    7. Use Advanced Blocks only for renderer-specific blocks such as Plotly, Mermaid, JSXGraph, Arquero, Manim, and SVG.
    8. Use Export / Diagnostics for JSON, Reveal export, and export-readiness checks.
    9. Add quiz questions after sections are stable.
    10. Use rich quiz stems only when the visual or computation is pedagogically necessary.
    11. Add print/PDF pagination controls where needed: page-break, avoid-break, and allow-break.
    12. Add Reveal slide breaks where a section is too dense for one slide.
    13. Export Active Lesson and inspect blockInventory.
    14. Re-import the export as a round-trip test.

    24. Printing, PDF Export, and Pagination Controls

    Use the Print button in the top bar for browser print or Save as PDF. The sidebar, guide, editor, calculator, statistics panel, and quiz controls are hidden in print mode. MathLesson applies print-safe contrast and tries to keep major blocks from being split across page boundaries.

    Author controls

    When a lesson is long, add explicit pagination markup directly in section HTML. These controls are preserved in exported JSON because they live inside sections[i].content.

    22.1 Force a new printed/PDF page

    Use <hr class="page-break"> where a new page should begin.

    <h2>Long Example</h2>
<p>First part of the explanation...</p>

<hr class="page-break">

<h2>Continuation</h2>
<p>Second part starts on a new printed page.</p>

    22.2 Prevent a block from splitting

    Use avoid-break around a group that should stay together: a paragraph plus equation, a callout, a small table, a figure caption, or a short worked example.

    <div class="avoid-break">
  <div class="callout ex">
    <div class="callout-tag">Example</div>
    <p>Keep this explanation and equation together.</p>
    <div class="math-display">\[PV = \sum_t v^t C_t\]</div>
  </div>
</div>

    22.3 Allow an oversized block to split

    Use allow-break when a block is naturally taller than one printed page. This prevents pathological blank-page behavior.

    <div class="allow-break">
  <table>
    ... very long table ...
  </table>
</div>

    22.4 Recommended pagination policy

    Content typeRecommended markupReason
    Short calloutavoid-breakDefinition, theorem, proof, or example should remain visually atomic.
    Small tableavoid-breakHeader and rows should not separate awkwardly.
    Long tableallow-breakA table taller than one page must be allowed to continue.
    Plotly / JSXGraph / SVG / Mermaidavoid-breakFigures should not be cut by page changes.
    New major topicpage-breakStart important sections cleanly on a new page.

    22.5 Reveal slide breaks

    Printed/PDF page breaks are separate from Reveal slide breaks. Use <hr class="slide-break"> or <!-- slide --> to split presentation slides.

    <p>First slide content.</p>
<hr class="slide-break">
<p>Second slide content.</p>

    22.6 Diagnostics

    When printing or exporting Reveal decks, MathLesson displays diagnostics for likely layout risks: long sections, large tables, many equations, multiple visual blocks, or missing slide breaks. Diagnostics are warnings, not proof of failure; inspect print preview or the exported deck before final use.

    30. v4.4.3 Comprehensive Block Tutorials

    This section is the authoring manual for the block system. The general rule is simple: the author inserts semantic blocks into sections[i].content, previews the section, then uses Reveal and Print/PDF planners to decide where to add explicit layout controls.

    Authoring invariant

    Every block should answer three questions: what is the mathematical purpose, how should it render on screen, and how should it behave when exported to Student HTML, Reveal, and Print/PDF?

    30.1 Prose and section structure

    Use plain paragraphs for exposition. Keep paragraphs short enough to survive web, phone, slide, and PDF layouts. A section should usually start with a heading, a short objective, then the main mathematical content.

    <h1>Curvature of a Plane Curve</h1>
<p class="quiz-meta">Differential Geometry ┬Ę ┬¦1</p>
<p>The goal is to connect velocity, acceleration, and curvature.</p>
    • Use when: writing ordinary explanations, transitions, objectives, or summaries.
    • Common mistake: one very long paragraph that becomes unreadable on phone or slides.
    • Export behavior: prose can split naturally unless wrapped in avoid-break.

    30.2 Inline math and display math

    Use inline math for small expressions inside prose. Use display math for equations that the learner must inspect as standalone objects.

    Inline: The speed is \(v(t)=\|r'(t)\|\).

<div class="math-display">
  <div class="math-label">Curvature</div>
  \[ \kappa(t)=\frac{\|r'(t)\times r''(t)\|}{\|r'(t)\|^3} \]
</div>
    • Use when: the formula is central to the lesson.
    • Common mistake: using display math for every small expression.
    • Reveal behavior: large display math may need its own slide.
    • Print/PDF behavior: wrap important equation groups in avoid-break.

    30.3 Callout blocks

    Callouts encode didactic intent. Use them to distinguish definitions, theorems, identities, remarks, examples, proofs, and exercises.

    ClassMeaningTypical use
    defDefinitionIntroduce a term or object.
    thmTheoremState a result to prove or apply.
    idIdentity / formulaEmphasize an algebraic formula.
    rmkRemarkGive nuance or warning.
    exExampleShow a worked application.
    prfProofGive an argument.
    exrExercisePrompt learner action.
     
    <div class="callout def avoid-break">
  <div class="callout-tag">Definition ŌĆö Unit tangent</div>
  <p>For a regular curve \(r(t)\), the unit tangent is</p>
  <div class="math-display">\[T(t)=\frac{r'(t)}{\|r'(t)\|}\]</div>
</div>
    • Use when: the learner should recognize a block's role immediately.
    • Common mistake: using one callout type for everything.
    • Print/PDF behavior: short callouts usually need avoid-break.

    30.4 Responsive grid blocks

    Grid blocks let the author place related material side by side while preserving responsive behavior in the author view and student export.

    <div class="ml-grid definition-example-grid" data-grid-type="definition-example-grid" data-grid-version="4.2.0">
  <div class="ml-grid-title">Definition and example</div>
  <div class="ml-grid-cell">
    <h3>Definition</h3>
    <p>Curvature measures turning per unit arclength.</p>
  </div>
  <div class="ml-grid-cell">
    <h3>Example</h3>
    <p>A circle of radius \(R\) has \(\kappa=1/R\).</p>
  </div>
</div>
    • Use when: two or three concepts must be compared, paired, or visually coordinated.
    • Common mistake: putting too many large interactive blocks into one grid.
    • Student export: should preserve the same author-defined grid layout subject to responsive width.
    • Reveal behavior: dense grids usually need a slide break before or after the grid.
    • Print/PDF behavior: wide grids may need print-landscape or a dedicated page.

    30.5 Plotly Explore 2D widgets

    Use a 2D Explore widget when the learner should manipulate parameters and see the curve change.

    <div class="explore-widget" data-explore="2d"
  data-title="Sine family"
  data-params='[{"id":"A","label":"A","min":0.1,"max":3,"step":0.1,"value":1}]'
  data-fn="(x,p)=>p.A*Math.sin(x)"
  data-xrange="[-6.28,6.28]"
  style="height:360px;"></div>
    • Use when: a parameter changes the shape of a curve.
    • Common mistake: malformed JSON in data-params.
    • Reveal behavior: one interactive plot per slide is safest.
    • Print/PDF behavior: printed output is static; explain what the slider demonstrates in surrounding prose.

    30.6 Plotly Explore 3D widgets

    Use a 3D Explore widget for surfaces or spatial examples. Keep the surface simple; 3D widgets are heavier than 2D widgets.

    <div class="explore-widget" data-explore="3d"
  data-title="Parameterized surface"
  data-params='[{"id":"a","label":"a","min":0,"max":2,"step":0.1,"value":1}]'
  data-fn="(x,y,p)=>Math.sin(x)*Math.cos(y)+p.a"
  data-xrange="[-3,3]" data-yrange="[-3,3]"
  style="height:420px;"></div>
    • Use when: the lesson requires a surface, not just a curve.
    • Common mistake: overloading one widget with too many sliders.
    • Export behavior: verify in student export and Reveal because 3D widgets require more layout space.

    30.7 Mermaid diagrams

    Use Mermaid for flowcharts, state diagrams, and conceptual dependencies. Keep syntax plain and labels simple.

    <div class="mermaid-wrap">
  <div class="mermaid-label">Concept flow</div>
  <pre class="mermaid">flowchart LR
    A[Velocity] --> B[Speed]
    B --> C[Unit tangent]
    C --> D[Curvature]
  </pre>
</div>
    • Use when: showing process, implication, dependency, or state flow.
    • Common mistake: complex labels with punctuation that Mermaid parses badly.
    • Print/PDF behavior: diagrams should usually be kept together with avoid-break.

    30.8 JSXGraph boards

    Use JSXGraph for interactive geometry. Put the board in a jsx-wrap and keep the initialization code close to the board.

    <div class="jsx-wrap" data-jsx="unit-circle" data-init="inline">
  <div class="jsx-header">Unit circle</div>
  <div class="jxgbox" style="height:320px;"></div>
  <script type="text/jsxgraph">
    const board = JXG.JSXGraph.initBoard(el, {
      boundingbox:[-2,2,2,-2], axis:true
    });
    board.create('circle', [[0,0], 1]);
  </script>
</div>
    • Use when: learners must drag, inspect, or construct geometric objects.
    • Common mistake: initializing the board before the panel has visible width; MathLesson delays initialization in student export.
    • Reveal behavior: one JSXGraph board per slide is safest.

    30.9 Arquero-style data/table blocks

    Use an Arquero-style block when data should be generated or transformed before display.

    <div class="aq-wrap" data-aq-title="Sample data">
  <script type="text/arquero">
    const tbl = aq.table({x:[1,2,3], y:[1,4,9]});
    return renderAqTable(tbl, "Squares");
  </script>
</div>
    • Use when: a table is better generated from data than hand-coded row by row.
    • Common mistake: forgetting to return HTML from the script.
    • Print/PDF behavior: long tables should use allow-break.

    30.10 Statistics/probability blocks

    Statistics blocks are reusable outputs from the Statistics panel. Use them for summaries, distributions, simulations, regressions, and survival/life-table examples.

    <div class="stat-block" data-stat-type="probability-distribution"
  data-stat-config='{"op":"distribution","dist":"normal","params":{"mean":0,"sd":1,"x":1.96}}'></div>
    • Use when: the lesson needs reproducible statistical output.
    • Common mistake: malformed JSON inside data-stat-config.
    • Export behavior: scan export planners because statistics blocks may contain both table and plot output.

    30.11 SVG blocks

    SVG is the most deterministic diagram format for print and export because it is inline and self-contained.

    <div class="svg-wrap" data-title="Triangle">
  <svg viewBox="0 0 160 120" role="img" aria-label="Triangle">
    <polygon points="20,100 140,100 80,20" fill="none" stroke="currentColor" stroke-width="3"/>
  </svg>
</div>
    • Use when: a static diagram must print cleanly.
    • Common mistake: omitting viewBox, which limits responsive scaling.
    • Print/PDF behavior: SVG usually exports better than canvas-based visuals.

    30.12 Manim/video placeholders

    Use Manim/video blocks for externally rendered animations. The video file must be in the same folder as the HTML file, or the src must point to a reachable relative or absolute path.

    <div class="manim-slot">
  <div class="manim-header">scene.mp4 ŌĆö Frenet frame animation</div>
  <video controls src="scene.mp4" preload="metadata" playsinline></video>
</div>
    • Use when: a concept requires animation that is better rendered outside the browser.
    • Common mistake: the video is not in the same folder as the HTML app or exported student file.
    • Student export: the video path must also be valid relative to the exported student HTML file.
    • Print/PDF behavior: video becomes a static black/media region; add prose explaining the animation.

    30.13 Exercise widgets and quiz items

    Use inline exercise widgets for local practice inside sections. Use the Quiz tab for formal assessment.

    <div class="callout exr avoid-break">
  <div class="callout-tag">Exercise</div>
  <p>Compute the curvature of a circle of radius \(R\).</p>
</div>

    Quiz items support multiple choice, true/false, math input, and free response. Math input should use expected LaTeX answers when exact symbolic grading is needed.

    {
  "type": "math",
  "stem": "Enter one fourth as a fraction.",
  "answer": "\\frac{1}{4}",
  "feedback": "Use the fraction form."
}
    • Use inline exercises when: practice belongs immediately after an example.
    • Use quiz items when: the learner should receive cumulative assessment.
    • Student export: quiz answers and scores persist locally in the exported student file.

    30.15 Authoring checklist for every block

    1. Identify the block's didactic purpose: definition, example, visual, computation, exercise, or assessment.
    2. Insert the smallest block that communicates that purpose.
    3. Preview immediately in the main lesson view.
    4. Check whether the block appears correctly in student export.
    5. Scan Reveal export if the lesson will become slides.
    6. Scan Print/PDF if the lesson will become a handout.
    7. Add avoid-break, allow-break, page-break, or <!-- slide --> only where the planner or preview shows a real need.
    8. Export JSON and re-import when doing major structural edits.
